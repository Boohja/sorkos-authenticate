Authentication Workflow

This is the intended authorization-code flow between a client project, Sorkos Login, and the selected OAuth provider.

1. Client Redirects To Sorkos Login

The client project creates a random state, stores it in its own local session, and redirects the user to /authorize.

{{ @request_origin }}/authorize?client_id=example-id&redirect_uri=https%3A%2F%2Fclient.example%2Fauth%2Fcallback&response_type=code&state=example-state&lang=en
Client redirect examples 
<?php
session_start();

$state = bin2hex(random_bytes(32));
$_SESSION['sorkos_oauth_state'] = $state;

$params = http_build_query([
    'client_id' => 'example-id',
    'redirect_uri' => 'https://client.example/auth/callback',
    'response_type' => 'code',
    'state' => $state,
    'lang' => 'en',
]);

header('Location: {{ @request_origin }}/authorize?' . $params);
exit;
import crypto from 'node:crypto';

app.get('/login', (req, res) => {
  const state = crypto.randomBytes(32).toString('hex');
  req.session.sorkosOAuthState = state;

  const params = new URLSearchParams({
    client_id: 'example-id',
    redirect_uri: 'https://client.example/auth/callback',
    response_type: 'code',
    state,
    lang: 'en'
  });

  res.redirect(`{{ @request_origin }}/authorize?${params.toString()}`);
});

2. Sorkos Login Validates The Request

Sorkos Login checks that the client exists, is active, supports the requested response type, and that the redirect URI is an exact registered match.

3. User Signs In With A Provider

If there is no central Sorkos Login session, the user chooses an enabled provider. The first implemented provider is E-Mail, which sends a six-digit one-time code through Sorkos Login's internal browser provider flow. Google and Discord remain configured placeholders until their external OAuth callbacks are implemented.

4. Sorkos Login Redirects Back To The Client

After successful authentication, Sorkos Login creates a short-lived authorization code and redirects the user back to the original client callback.

https://client.example/auth/callback?code=example-code&state=example-state
Callback validation examples 
<?php
session_start();

$returnedState = $_GET['state'] ?? '';
$expectedState = $_SESSION['sorkos_oauth_state'] ?? '';

if ($returnedState === '' || !hash_equals($expectedState, $returnedState)) {
    http_response_code(400);
    exit('Invalid state');
}

$code = $_GET['code'] ?? '';
app.get('/auth/callback', (req, res, next) => {
  const returnedState = req.query.state || '';
  const expectedState = req.session.sorkosOAuthState || '';

  if (!returnedState || returnedState !== expectedState) {
    res.status(400).send('Invalid state');
    return;
  }

  const code = req.query.code || '';
  next();
});

5. Client Exchanges The Code Server-Side

The client validates the returned state, then posts the authorization code to /token from its server. If valid, Sorkos Login returns normalized user information.

Token exchange examples 
<?php
$response = file_get_contents('{{ @request_origin }}/token', false, stream_context_create([
    'http' => [
        'method' => 'POST',
        'header' => "Content-Type: application/x-www-form-urlencoded\r\n",
        'content' => http_build_query([
            'grant_type' => 'authorization_code',
            'code' => $code,
            'client_id' => 'example-id',
            'client_secret' => 'example-secret',
            'redirect_uri' => 'https://client.example/auth/callback',
        ]),
    ],
]));

$tokenResult = json_decode($response, true);
const response = await fetch('{{ @request_origin }}/token', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/x-www-form-urlencoded'
  },
  body: new URLSearchParams({
    grant_type: 'authorization_code',
    code,
    client_id: 'example-id',
    client_secret: 'example-secret',
    redirect_uri: 'https://client.example/auth/callback'
  })
});

const tokenResult = await response.json();

6. Client Creates Its Own Local Session

The client stores its own local session data. Sorkos Login does not share cookies across projects and does not write sessions for client applications.

Current implementation status: /authorize validation, the provider-selection screen, and first-party E-Mail login exist. External OAuth providers, authorization-code issuing, /token, and /logout are still upcoming implementation steps.
